# ЧАСТИНА 1. Покрокова інструкція

## Крок 1. Репозиторій на GitHub
1. Створи новий репозиторій (напр. `it-blog`).
2. Розпакуй архів проєкту, поклади вміст у теку репозиторію.
3. Залий код:
   ```bash
   git init
   git add .
   git commit -m "Initial commit: IT Blog (Next.js + Express + PostgreSQL)"
   git branch -M main
   git remote add origin https://github.com/<твій-логін>/it-blog.git
   git push -u origin main
   ```
   `README.md` уже є в корені — вимога завдання виконана.

> Перед пушем переконайся, що проєкт запускається локально (інструкція в `README.md`, розділ «Локальний запуск»).

## Крок 2. Розгортання на Railway (3 сервіси)

> ⚠️ **Важливо про оплату (станом на 2026):** у Railway більше немає постійного безкоштовного тарифу.
> Новий акаунт отримує разовий **trial-кредит $5 на 30 днів** — цього з головою вистачає, щоб зробити деплой,
> скріншоти та налаштувати GSC. Зроби лабу за один підхід, щоб не витрачати кредит даремно.
> Якщо кредит вичерпається — лаба дозволяє використати стандартний домен `*.railway.app` замість `.pp.ua`.

**2.1. PostgreSQL**
- `New Project → Deploy PostgreSQL` (або всередині проєкту `+ New → Database → PostgreSQL`).
- Railway автоматично створить змінну `DATABASE_URL`.

**2.2. Backend**
- У тому ж проєкті: `+ New → GitHub Repo` → обери свій репозиторій.
- Відкрий сервіс → **Settings → Root Directory:** `apps/backend`.
- **Settings → Networking → Generate Domain** (отримаєш публічний URL бекенду).
- **Variables** (додай):
  - `DATABASE_URL` → натисни «Reference» і обери змінну з сервісу Postgres.
  - `JWT_SECRET` → будь-який довгий випадковий рядок.
  - `PGSSL` → `true`
  - `CORS_ORIGIN` → `*` (тимчасово; потім зміниш на домен фронтенду).
- Дочекайся успішного deploy.

**2.3. Застосувати міграції (один раз)**
- Відкрий сервіс Postgres → вкладка **Variables/Connect** → скопіюй **публічний** `DATABASE_URL` (Public Network).
- Локально:
  ```bash
  cd apps/backend
  DATABASE_URL="<публічний-URL>" PGSSL=true npm run migrate -- --seed
  ```
- Перевір, що бекенд віддає дані: відкрий `https://<твій-backend>.up.railway.app/api/articles` — має бути JSON.

**2.4. Frontend**
- `+ New → GitHub Repo` → той самий репозиторій (так, ще раз).
- **Settings → Root Directory:** `apps/frontend`.
- **Settings → Networking → Generate Domain** (це і буде URL сайту, поки немає власного домену).
- **Variables**:
  - `NEXT_PUBLIC_API_URL` → публічний URL бекенду (`https://<backend>.up.railway.app`).
  - `API_URL` → той самий URL бекенду.
  - `NEXT_PUBLIC_SITE_URL` → публічний URL фронтенду (або майбутній домен `.pp.ua`).
- Дочекайся deploy і **відкрий сайт за публічним URL**. Маєш побачити головну з 12 статтями.

✅ **Зафіксуй:** URL фронтенду, URL бекенду, скріншот успішного deploy.

## Крок 3. Реєстрація домену `.pp.ua`
1. Зайди на [nic.ua](https://nic.ua), зареєструйся.
2. Перевір і зареєструй [безкоштовний домен](https://nic.ua/uk/domains/.pp.ua) `.pp.ua`, напр. `myitblog.pp.ua`.
3. Зафіксуй: ім'я домену, NS-сервери, дату реєстрації.

> Якщо `.pp.ua` недоступний/не виходить — використовуй `*.up.railway.app` для решти кроків (лаба це дозволяє).

## Крок 4. Підключення домену до Railway
1. У сервісі **frontend**: `Settings → Networking → Custom Domain` → введи `myitblog.pp.ua`.
2. Railway покаже **CNAME-значення** (вигляду `xxxx.up.railway.app`) — скопіюй.
3. На nic.ua у DNS-налаштуваннях домену додай запис:
   ```
   Тип:      CNAME
   Ім'я:     @  (або www)
   Значення: <значення, яке дав Railway>
   ```
   > Примітка: для кореня домену (`@`) деякі реєстратори вимагають ALIAS/ANAME замість CNAME.
   > Якщо nic.ua не дає поставити CNAME на `@`, став CNAME на `www`, а корінь переадресуй на `www`.
4. Онови `NEXT_PUBLIC_SITE_URL` у Railway на `https://myitblog.pp.ua` і передеплой.
5. Дочекайся **DNS propagation** (5 хв – 24 год). Перевір на [dnschecker.org](https://dnschecker.org).

## Крок 5. Дослідження «Що бачить Google»

**5.1. curl-запит** (підстав свій URL):
```bash
curl https://myitblog.pp.ua > curl-result.txt
# або, якщо без домену:
curl https://<твій-frontend>.up.railway.app > curl-result.txt
```

**5.2. Аналіз результату.** Відкрий `curl-result.txt`. Оскільки наш сайт — **SSR**, у HTML буде:
- повний текст/заголовки статей (шукай, напр., «Docker у 2026», «Як працюють великі мовні моделі»);
- тег `<title>` з назвою сайту;
- `<meta name="description" ...>`;
- заповнений `<body>`.

Заповни таблицю (приклад уже коректний для нашого сайту — звір зі своїм файлом):

| Елемент | Присутній | Що містить |
|---|---|---|
| Текст статей | **Так** | заголовки й анонси статей прямо в HTML |
| `<title>` | **Так** | `IT Blog — Новини світу інформаційних технологій` |
| `<meta name="description">` | **Так** | опис сайту для пошукових систем |
| Вміст `<body>` | — | повний контент сторінки (хедер, картки статей, футер), а не порожній контейнер |

**5.3. View Page Source у браузері.**
- Відкрий сайт → `ПКМ → Переглянути код сторінки (View Page Source)`.
- Порівняй із `curl-result.txt`: для SSR вони **майже ідентичні** (обидва містять готовий HTML).
- Це і є різниця з CSR: там View Source був би майже порожнім, а контент з'являвся б лише в DevTools (DOM).

**5.4. Перевірка індексації в Google.** У пошуку Google введи:
```
site:myitblog.pp.ua
```
Спочатку нічого не буде (сайт новий) — це нормально. Зафіксуй скріншот (порожній результат), а після Кроку 7 можна перевірити повторно через кілька днів.

## Крок 6. Google Search Console (верифікація через DNS TXT)
1. Відкрий [search.google.com/search-console](https://search.google.com/search-console).
2. `Add Property → Domain` → введи `myitblog.pp.ua`.
3. GSC покаже **TXT-запис** виду `google-site-verification=xxxxxxxx`.
4. На nic.ua додай у DNS:
   ```
   Тип:      TXT
   Ім'я:     @
   Значення: google-site-verification=xxxxxxxx
   ```
5. Зачекай 5–30 хв (propagation) → натисни **Verify** у GSC.
6. Зафіксуй скріншот успішної верифікації.

> Якщо використовуєш `*.railway.app` (без власного домену) — обирай у GSC верифікацію через **HTML-тег** або **HTML-файл**
> (DNS до railway.app ти не контролюєш). Для HTML-тегу можна тимчасово додати `<meta>` у `app/layout.js`,
> або поклади HTML-файл у `apps/frontend/public/` — Railway віддасть його за коренем сайту.

## Крок 7. Перший запит на індексацію
1. У GSC → **URL Inspection** → введи головну сторінку сайту.
2. Натисни **Request Indexing**.
3. Зафіксуй скріншот.

Скопіюй цей блок у свій звіт і заповни `[...]`.

```
ЗВІТ З ЛАБОРАТОРНОЇ РОБОТИ №1
Студент: [ПІБ, група]

1. URL розгорнутого сайту на Railway: [https://...up.railway.app]
2. Назва зареєстрованого домену:       [myitblog.pp.ua]
3. Скріншот успішного deploy:           [вставити скріншот]
4. Вміст curl-result.txt + пояснення:   [вставити фрагмент HTML + текст нижче]
5. Порівняльна таблиця curl / View Source / DevTools: [таблиця нижче]
6. Скріншот верифікації в GSC:          [вставити скріншот]
7. Скріншот запиту на індексацію:       [вставити скріншот]
8. Відповідь: "Що побачить Google crawler і чому це може бути проблемою?": [текст нижче]
```

**До п.4 (пояснення curl).**
> `curl` робить простий HTTP-запит без виконання JavaScript — тобто бачить рівно те, що віддає сервер.
> Оскільки сайт побудований на Next.js із серверним рендерингом, сервер повертає **повністю сформований HTML**:
> заголовки та анонси статей, теги `<title>` і `<meta description>`, структуру сторінки. Саме цей HTML і отримує
> пошуковий робот на першому етапі. Тому контент сайту доступний для індексації одразу.

**До п.5 (порівняльна таблиця).**

| Метод | Що показує | Виконує JS? | Результат для нашого сайту |
|---|---|---|---|
| `curl` | сирий HTML від сервера | ні | повний контент (SSR) |
| View Page Source | сирий HTML від сервера | ні | повний контент, ≈ як у curl |
| DevTools (Elements/DOM) | підсумковий DOM у браузері | так | повний контент (+ результат гідратації React) |

> Висновок: для SSR усі три методи показують контент. Для CSR-сайту `curl` і View Source були б майже порожні,
> а контент з'явився б лише в DevTools після виконання JS.

**До п.8 (що побачить Google crawler).**
> Google crawler побачить **готовий HTML із повним текстом статей, унікальними `<title>`/`<meta description>`,
> мікророзміткою JSON-LD (Article), а також `sitemap.xml` і `robots.txt`**. Для нашого сайту це **не є проблемою** —
> навпаки, це бажаний стан: контент індексується одразу, без залежності від виконання JavaScript.
> Проблемою це було б для CSR-застосунку (чистий React/Vue без SSR): там краулер на першому проході отримав би
> порожній `<div id="root">`, а індексація контенту відклалася б до другої «хвилі» рендерингу (або не відбулася зовсім).
> Саме тому ТЗ вимагає SSR-фреймворк — і ми його використали.

# ЧАСТИНА 3. Відповіді на контрольні питання

## Рівень 1 — терміни

**1. Що таке SEO і чим відрізняється від платної реклами (SEA)?**
SEO (Search Engine Optimization) — оптимізація сайту для **органічної** (безкоштовної) видачі: технічні покращення,
контент, структура, посилання, щоб сайт ранжувався вище. SEA (Search Engine Advertising) — **платні** оголошення
(напр. Google Ads), що показуються над/під органічною видачею з позначкою «Реклама». Відмінності: SEO дає результат
повільніше, але «безкоштовний» трафік триває довго; SEA дає трафік миттєво, але лише поки ти платиш за кліки.

**2. Різниця між crawling, indexing та ranking. Аналогія.**
- **Crawling (сканування):** робот (Googlebot) обходить сайт за посиланнями й завантажує сторінки.
- **Indexing (індексування):** проаналізований вміст зберігається в індексі пошуковика.
- **Ranking (ранжування):** при запиті користувача пошуковик обирає й упорядковує сторінки з індексу за релевантністю.

*Аналогія з бібліотекою:* crawling — бібліотекар ходить між полицями й переглядає всі книжки; indexing — він заносить
кожну книжку в каталог із ключовими словами; ranking — коли відвідувач питає тему, бібліотекар вирішує, яку книжку
порадити першою.

**3. Що таке DNS і його роль при відкритті сайту?**
DNS (Domain Name System) — «телефонна книга» інтернету: перетворює доменне ім'я (`myitblog.pp.ua`) на IP-адресу
сервера. Коли ти вводиш домен у браузер, відбувається DNS-запит → отримання IP → лише після цього браузер
встановлює з'єднання із сервером за цією адресою.

**4. Що таке CNAME і чим відрізняється від A-запису?**
- **A-запис** зіставляє домен із **IPv4-адресою** (напр. `93.184.216.34`).
- **CNAME (Canonical Name)** — це **псевдонім**: вказує, що домен «дорівнює» іншому **доменному імені**
  (напр. `myitblog.pp.ua → xxxx.up.railway.app`), і вже той дозволяється у IP.
CNAME зручний, коли цільова IP може змінюватися (як у хмарних хостингів): ти посилаєшся на ім'я, а не на адресу.
Обмеження: класичний CNAME не можна ставити на кореневий домен одночасно з іншими записами (звідси ALIAS/ANAME).

**5. Навіщо потрібен TXT-запис? Інші завдання, крім верифікації GSC.**
TXT зберігає довільний текст у DNS. Окрім підтвердження прав на домен (Google, Microsoft тощо), його використовують для:
- **SPF** — список серверів, яким дозволено надсилати пошту від домену;
- **DKIM** — публічний ключ для цифрового підпису листів;
- **DMARC** — політика обробки листів, що не пройшли SPF/DKIM;
- верифікації інших сервісів (SSL-центри, хостинги, поштові сервіси).

## Рівень 2 — аналіз

**6. `curl` показав лише `<div id="root"></div>`. Чому і що це означає для пошуку?**
Це ознака **CSR (Client-Side Rendering)**: сервер віддає порожній HTML-каркас, а весь контент малює JavaScript уже
в браузері. `curl` (і пошуковий робот на першому проході) JS не виконує → бачить порожній контейнер. Наслідок для SEO:
контент може бути не проіндексований або індексуватися із затримкою, метатеги відсутні, ранжування погіршується.
*(У нашому SSR-проєкті такої проблеми немає — `curl` бачить повний HTML.)*

**7. Чим View Page Source відрізняється від DOM у DevTools? Чому це важливо для SEO?**
- **View Page Source** — це **сирий HTML**, який віддав сервер (до виконання JS).
- **DevTools → Elements** — це **підсумковий DOM** після виконання JS і змін на сторінці.
Для SEO важливо, бо пошуковий робот спочатку працює із сирим HTML (як View Source). Якщо контент з'являється лише
в DOM (через JS), на першому проході його може не бути в індексі. Тому критичний контент має бути вже в HTML від сервера.

**8. Що таке DNS propagation і чому зміни не миттєві?**
DNS-записи кешуються на безлічі серверів (провайдери, резолвери, ОС) на час **TTL**. Після зміни запису старе значення
ще «живе» в кешах, поки не спливе TTL і вони не оновляться. Поширення (propagation) по світу займає від кількох хвилин
до 24–48 годин. Тому новий CNAME/TXT застосовується не одразу.

**9. White / Grey / Black SEO — різниця і приклади.**
- **White hat** — методи в межах правил пошуковиків: якісний унікальний контент, коректні метатеги, семантична
  розмітка, швидкість, природні посилання, зручна структура.
- **Grey hat** — на межі правил: масова закупівля посилань низької якості, надмірна оптимізація під ключі,
  «прокручені» поведінкові фактори, агресивний обмін посиланнями.
- **Black hat** — заборонені маніпуляції: прихований текст/посилання (cloaking), keyword stuffing, дорвеї,
  спам у коментарях, PBN-мережі, накрутка. Ризик — санкції та виключення з індексу.

**10. Чому GSC вимагає підтвердження власника і які способи верифікації?**
Щоб лише власник міг бачити приватні дані сайту (запити, помилки індексації) і керувати ним. Способи:
**DNS TXT-запис**, **HTML-файл** у корені сайту, **HTML-метатег** на головній, через **Google Analytics** або
**Google Tag Manager**. Підтвердження прав на рівні домену (DNS) покриває всі піддомени й протоколи одразу.

## Рівень 3 — синтез

**11. Чи готовий наш сайт до індексації? Обґрунтуй.**
**Так.** Аргументи:
- **SSR/SSG** — повний HTML із контентом віддається сервером (підтверджено `curl`-перевіркою).
- **Унікальні `<title>` та `<meta description>`** для кожної сторінки; є **Open Graph** і **canonical**.
- **JSON-LD (Article)** — структуровані дані для статей.
- **`sitemap.xml`** із усіма статтями/категоріями/авторами/тегами та **`robots.txt`** (адмінка й пошук закриті).
- Семантична HTML-структура, оптимізовані зображення через `next/image`.
Залишається лише розгорнути, додати в GSC і надіслати на індексацію — тобто кроки 2–7 цієї лаби.

**12. Googlebot вміє виконувати JS, але з CSR усе одно є проблема. Чому?**
- **Дві хвилі індексації:** спершу індексується сирий HTML; рендеринг JS відкладається в чергу й може запізнюватися
  (години–дні), тож свіжий контент індексується повільніше.
- **Render budget / ліміти ресурсів:** виконання JS дороге; для великих сайтів бюджет рендерингу обмежений.
- **Крихкість:** помилка JS, повільний/заблокований скрипт, залежність від API, таймаут — і контент не відрендериться.
- **Інші боти** (Bing, соцмережі для прев'ю, частина краулерів) гірше або взагалі не виконують JS.
Тому навіть із виконанням JS у Google повний CSR залишається ризикованим для SEO.

**13. Три зміни без переходу на SSR, щоб покращити індексацію CSR-сайту.**
1. **Pre-rendering / dynamic rendering:** віддавати ботам заздалегідь відрендерену HTML-версію
   (статичний експорт ключових сторінок або сервіс типу prerender.io / Rendertron).
2. **Серверні метатеги:** генерувати унікальні `<title>`, `<meta description>`, Open Graph і canonical уже на сервері/едж
   (а не лише через JS), щоб робот бачив їх у сирому HTML.
3. **`sitemap.xml` + якісне внутрішнє перелінкування** (і коректний `robots.txt`), щоб робот швидше знаходив усі URL;
   додатково — критичний контент у `<noscript>` як запасний варіант.

**14. DNS TXT vs HTML-файл для верифікації в GSC: переваги й недоліки.**
- **DNS TXT:**
  - *Плюси:* підтверджує права на **весь домен** (усі піддомени й протоколи), не залежить від хостингу/деплоїв,
    запис лишається назавжди.
  - *Мінуси:* потрібен доступ до DNS-панелі; діє після propagation (не миттєво); незручно, якщо домен не твій.
- **HTML-файл (або метатег):**
  - *Плюси:* швидко й просто, якщо є доступ до файлів сайту; не треба чіпати DNS; підтверджується одразу.
  - *Мінуси:* верифікує лише **конкретний хост/URL** (не весь домен); файл можна випадково видалити при передеплої;
    треба, щоб хостинг віддавав статичний файл за коренем.
Для власного домену зручніший **DNS TXT**; для `*.railway.app` (де DNS не твій) — **HTML-файл/метатег**.
